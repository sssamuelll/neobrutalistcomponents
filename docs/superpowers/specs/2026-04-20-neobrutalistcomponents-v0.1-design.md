# neobrutalistcomponents · v0.1.0 Design

**Date:** 2026-04-20
**Owner:** Samuel Ballesteros (SDB)
**Status:** Approved for implementation planning

---

## 1. Summary

Relaunch `neobrutalistcomponents` from its dormant `v0.0.1` prototype (2 partial components) into a publishable v0.1.0 React component library with a distinctive, multi-flavor identity.

The v0.1 release ships **3 deeply-refined components** (`Button`, `Input`, `Card`) and **4 swappable visual themes** (`classic`, `tech`, `swiss`, `y2k`) driven by a CSS-variable token system. A custom single-page landing with a live theme switcher serves as both documentation and portfolio piece.

---

## 2. Goals & non-goals

### Goals
- Publish to npm under `neobrutalistcomponents@0.1.0` as a genuinely usable library.
- Portfolio-worthy presentation: custom landing at GitHub Pages with a **live theme switcher** that re-renders the entire page into any of the 4 themes.
- Keep the library surface tight: 3 components shipped *well*, not 10 shipped poorly.
- Extensibility: a consumer can define a 5th theme in their own CSS without forking the library.

### Non-goals (explicitly out of scope for v0.1)
- Additional components (Select, Modal, Checkbox, Switch, Tabs, Tooltip, Toast, Table).
- `Textarea` — if ever added, it will be its own component, *not* a `multiline` prop on `Input`.
- Animations / motion system — theme changes are instantaneous; only simple hover states.
- Dark mode as a separate axis — each theme ships with its own color identity (e.g., `tech` is dark, others are light). No runtime `prefers-color-scheme` toggle.
- React 18 support — React 19 only. `ref` is a normal prop; no `forwardRef`.
- i18n of library-provided strings — the library has essentially none; consumer provides `errorMessage`, etc.
- `asChild` Radix-Slot pattern — good candidate for v0.2.
- Bundled icon library — components accept icon nodes via slot props; consumer brings their own.
- Visual regression testing — too expensive for a solo-maintained library; manual QA via the landing playground is sufficient.

```
  site/                       # landing + playground (NOT in library bundle)

  main.tsx                    # mounts <App/> for the site
```

### 3.2 Layering (consumer's perspective)

Five layers, strictly top-down:

1. **Consumer app** — imports components + theme CSS, sets `data-theme` attribute.
2. **Theme CSS** — one per theme. A scoped selector (`[data-theme="classic"]`) sets all `--nbc-*` custom properties.
3. **Component CSS** — reads tokens only via `var(--nbc-*)`. Never hardcodes color/font/border.
4. **React component** — exports typed API, composes DOM. No style logic in TypeScript.
5. **NeoProvider (optional)** — trivial wrapper rendering `<div data-theme={theme}>`, plus a `useTheme()` hook.

### 3.3 Token flow

```
Consumer: <div data-theme="classic"> <Button/> </div>
          │
          ▼
Theme CSS: [data-theme="classic"] {
             --nbc-fg: #111;
             --nbc-primary: #ffd500;
             --nbc-border: 3px solid #111;
             --nbc-shadow: 5px 5px 0 #111;
           }
          │
          ▼
Component CSS: .nbc-button {
                 color: var(--nbc-fg);
                 background: var(--nbc-primary);
                 border: var(--nbc-border);
                 box-shadow: var(--nbc-shadow);
               }
```

## 4. Theming system

### 4.1 Two-level token model

**Level 1 — `tokens.css` (base, always loaded):** theme-agnostic values plus safe defaults.

```css
:root {
  /* Spacing — invariant across themes */
  --nbc-space-xs: 4px;  --nbc-space-sm: 8px;
  --nbc-space-md: 12px; --nbc-space-lg: 18px;
  --nbc-space-xl: 24px;

  /* Font sizes — invariant */
  --nbc-fs-xs: 12px; --nbc-fs-sm: 14px;
  --nbc-fs-md: 16px; --nbc-fs-lg: 18px;

  /* Safe defaults (theme files override) */
  --nbc-font-sans: system-ui, sans-serif;
  --nbc-fg: #111; --nbc-bg: #fff;
  /* ... */
}
```

**Level 2 — `themes/*.css` (opt-in per-consumer import):** each theme redefines the full token contract under a scoped selector.

### 4.2 Theme token contract

Every theme must set:

**Typography**
- `--nbc-font-sans` — primary font stack
- `--nbc-font-mono` — monospace
- `--nbc-weight-label` — weight of labels / button text
- `--nbc-label-transform` — `none` | `uppercase`
- `--nbc-label-spacing` — letter-spacing of labels

**Color (semantic, not literal)**
- `--nbc-fg`, `--nbc-fg-muted`
- `--nbc-bg`, `--nbc-surface`
- `--nbc-primary`, `--nbc-primary-fg`
- `--nbc-danger`, `--nbc-danger-fg`
- `--nbc-border-color`
- `--nbc-focus` — focus ring color

**Shape**
- `--nbc-border-width` — 1–3 px
- `--nbc-border-style` — `solid` | `dashed`
- `--nbc-radius` — `0` | `14px` (only `y2k` uses non-zero)

**Effects**
- `--nbc-shadow` — default offset shadow
- `--nbc-shadow-lg` — for elevated surfaces
- `--nbc-rotate` — `0deg` | `-1.5deg` (`y2k` accent only)

### 4.3 Conventions

**Gradient tokens.** The `background` property accepts both colors and gradients, so a token like `--nbc-primary: linear-gradient(135deg, #ff6ec7, #ffd86b)` works without any component change. `y2k` uses this.

**Component-level sub-tokens.** Each component may declare its own tokens that default to the global ones but can be overridden at any scope:

```css
.nbc-button {
  --nbc-button-shadow: var(--nbc-shadow);
  box-shadow: var(--nbc-button-shadow);
}
.nbc-button:hover { --nbc-button-shadow: 2px 2px 0 var(--nbc-border-color); }
```

**Fifth-theme extensibility.** A consumer defines their own theme by writing CSS:

```css
[data-theme="minimal"] {
  --nbc-font-sans: 'IBM Plex Sans', sans-serif;
  --nbc-primary: #000;
  --nbc-border-width: 1px;
  --nbc-shadow: none;
  /* ...the rest of the contract */
}
```

No library fork required. This is part of the v0.1 marketing story.

### 4.4 Theme identity summary

| Theme | Fonts | Borders | Shadows | Color character |
| --- | --- | --- | --- | --- |
| `classic` | Inter, weight 800 | 3 px solid black | hard offset `5 5 0 #111` | bright flats (yellow / white / red) |
| `tech` | JetBrains Mono, uppercase | 1 px solid neon | neon glow + offset | dark bg, `#00ff88` accents |
| `swiss` | Helvetica, uppercase + wide spacing | 2 px solid black | *none* | b&w, editorial restraint |
| `y2k` | Comic Sans, `-1.5deg` rotate | 3 px dashed purple | multi-layer + inner | pastel gradient `primary`, pink/yellow |

## 5. Components API

All components extend their native DOM prop set (never wrap or hide standard attributes). All are SSR-safe (no `window` refs, no `useLayoutEffect` on the server). React 19 native `ref` (no `forwardRef`).

### 5.1 `Button`

```tsx
type ButtonVariant = 'primary' | 'secondary' | 'danger' | 'ghost';
type ButtonSize = 'sm' | 'md' | 'lg';

interface ButtonProps
  extends Omit<React.ButtonHTMLAttributes<HTMLButtonElement>, 'children'> {
  variant?: ButtonVariant;    // default 'primary'
  size?: ButtonSize;           // default 'md'
  loading?: boolean;           // spinner + aria-busy + disabled
  fullWidth?: boolean;
  leftIcon?: React.ReactNode;  // BYO icon slot
  rightIcon?: React.ReactNode;
  children?: React.ReactNode;
}
```

**Behaviour**
- `loading` sets `aria-busy="true"`, adds a spinner adornment, and disables interaction.
- `focus-visible` styling rendered via `--nbc-focus` token (never a generic outline).
- Icon slots wrap in a `<span aria-hidden="true">` so screen readers ignore them; the `children` text carries the label.
- Icon-only buttons: consumer passes only an icon in `children` and supplies `aria-label`.

### 5.2 `Input`

```tsx
type InputVariant = 'default' | 'error';
type InputSize = 'sm' | 'md' | 'lg';

interface InputProps
  extends Omit<React.InputHTMLAttributes<HTMLInputElement>, 'size'> {
  variant?: InputVariant;         // default 'default'
  size?: InputSize;                // default 'md'
  label?: React.ReactNode;
  helperText?: React.ReactNode;
  errorMessage?: React.ReactNode;
  leftIcon?: React.ReactNode;
  rightIcon?: React.ReactNode;
}
```

**Behaviour**
- Auto-generates `id` when not provided. `<label htmlFor>` and `aria-describedby` wire to the generated id.
- Auto-sets `aria-invalid="true"` when `variant === 'error'`.
- Rendering rules for helper/error text:
  - `variant='error'` + `errorMessage` set → renders `errorMessage` in error style; `helperText` (if also set) is hidden.
  - `variant='error'` + no `errorMessage` → no text rendered, but the input still shows error styling and `aria-invalid='true'`.
  - `variant='default'` + `helperText` set → renders `helperText` in muted style. `errorMessage` (if also set) is ignored.
  - `aria-describedby` always points at whichever text is actually rendered.

### 5.3 `Card` (composable)

```tsx
type CardVariant = 'default' | 'elevated' | 'interactive';

interface CardProps extends React.HTMLAttributes<HTMLDivElement> {
  variant?: CardVariant;
}

const Card: FC<CardProps> & {
  Header: FC<HTMLAttributes<HTMLDivElement>>;
  Title: FC<HTMLAttributes<HTMLHeadingElement>>;
  Description: FC<HTMLAttributes<HTMLParagraphElement>>;
  Content: FC<HTMLAttributes<HTMLDivElement>>;
  Footer: FC<HTMLAttributes<HTMLDivElement>>;
};
```

**Usage**

```tsx
<Card variant="elevated">
  <Card.Header>
    <Card.Title>Plan Pro</Card.Title>
    <Card.Description>$29/mo</Card.Description>
  </Card.Header>
  <Card.Content>Everything in free, plus…</Card.Content>
  <Card.Footer><Button>Subscribe</Button></Card.Footer>
</Card>
```

**Behaviour**
- `interactive` variant only adds `cursor: pointer` and hover styles. It does **not** change the element to `<button>` or add keyboard handlers. Consumers needing actual click behaviour wrap the card with `<a>` or `<button>` themselves.
- `Card.Title` always renders an `<h3>`. Consumers who need a different heading level skip `Card.Title` entirely and place their own `<h2>`/`<h4>`/etc. inside `Card.Header`.

### 5.4 `NeoProvider` + `useTheme`

```tsx
type NeoTheme = 'classic' | 'tech' | 'swiss' | 'y2k';

interface NeoProviderProps {
  theme: NeoTheme;                    // controlled by the consumer
  children: React.ReactNode;
  as?: keyof JSX.IntrinsicElements;   // default 'div'
  className?: string;
}

function useTheme(): NeoTheme | undefined;
```

- `NeoProvider` renders `<as data-theme={theme}>` and publishes `theme` via React context.
- `useTheme()` returns the nearest provider's theme, or `undefined` if used outside a provider (the consumer may be using `data-theme` directly).
- Theme changes are consumer-managed: they keep `useState<NeoTheme>` and pass the new value to `<NeoProvider theme={...}>`. The library does not own theme state.

## 6. Landing + playground

Single-page site at `src/site/`, built separately from the library. Deployed to GitHub Pages.

### 6.1 Sections

1. **Nav (sticky)** — logo / version chip / 4 theme chips acting as the global theme switcher / GitHub link.
2. **Hero** — large title, tagline, install command, view-source CTA, and an inline live `Button` demo.
3. **"The feature"** — short section explaining the 4-theme system with swatches.
4. **Button showcase** — all variants × sizes × states (including hover/loading/disabled/focus-visible/fullWidth/icons), with copy-paste JSX beside each.
5. **Input showcase** — all 3 sizes, default / focus / error / with-icons / with-label+helper / with-errorMessage.
6. **Card showcase** — 3 variants side by side with full composable markup.
7. **Bring-your-own-theme** — code block demonstrating a 5th theme under `[data-theme="minimal"]`.
8. **Quick start** — install + minimal usage.
9. **Footer** — MIT · author · GitHub · npm · docs links.

### 6.2 Live theme switcher

The nav's 4 chips map to `NeoTheme`. The root site component:

```tsx
function App() {
  const [theme, setTheme] = useState<NeoTheme>('classic');
  return (
    <NeoProvider theme={theme}>
      <Nav current={theme} onChange={setTheme} />
      {/* all showcase sections */}
    </NeoProvider>
  );
}
```

All 4 theme CSS files are imported upfront in `main.tsx` (~2.4 kB total; no lazy-load).

### 6.3 Deployment

GitHub Actions workflow (`deploy-pages.yml`) triggers on push to `main`, builds `site-dist/`, and publishes via GitHub's native `actions/deploy-pages@v4` flow (no `gh-pages` branch; uses the Pages artifact API).

## 7. Build & distribution

### 7.1 Vite configs

- `vite.config.ts` — library mode (existing, adapted). Entry `src/lib/index.ts`, outputs to `dist/`. Externalizes `react`, `react-dom`. `vite-plugin-dts` writes `.d.ts`.
- `vite.site.config.ts` — application mode. Entry `src/main.tsx`, outputs to `site-dist/`.

### 7.2 `package.json` (target state)

```json
{
  "name": "neobrutalistcomponents",
  "version": "0.1.0",
  "license": "MIT",
  "type": "module",
  "files": ["dist"],

    "dev": "vite --config vite.site.config.ts",
```

The current self-referencing `dependency` block is removed. `main`/`module` legacy fields are dropped in favor of `exports`.

### 7.3 TypeScript

- `strict: true` across the board.
- `tsconfig.app.json` covers `src/lib/`; site has its own inclusion or shares. No loose `any`.

## 8. Testing

### 8.1 Stack
- **Vitest** (jsdom environment)
- **@testing-library/react**
- **@axe-core/react** or `jest-axe`-equivalent for Vitest

### 8.2 What gets tested
- **Render smoke tests** — component renders for every `variant` × `size`.
- **Behaviour** — `loading` disables Button, `onClick` fires only when not loading/disabled, `errorMessage` renders + `aria-invalid` set when `variant='error'`.
- **Prop forwarding** — arbitrary DOM props (`data-*`, `aria-*`) land on the underlying element.
- **Accessibility** — axe runs against each component's primary render; zero violations required.
- **Provider** — `NeoProvider` sets `data-theme`, `useTheme` returns the theme; returns `undefined` outside any provider.

Site code (`src/site/`) is *not* covered by tests — it is the portfolio surface, validated manually via `npm run dev`.

Coverage target: **≥ 80 %** on `src/lib/`. No hard gate in CI; coverage is reported but not enforced.

## 9. CI / release

Three workflows in `.github/workflows/`:

1. **`ci.yml`** — runs on PR. `lint` → `typecheck` → `test` → `build:lib` → `build:site`. Must pass to merge.
2. **`deploy-pages.yml`** — runs on push to `main`. Builds `site` and publishes to GitHub Pages.
3. **`publish.yml`** — runs on tag `v*`. Runs the full CI, then `npm publish` using an `NPM_TOKEN` secret. Manual tagging only (no auto-versioning for v0.1).

## 10. Migration plan (6 phases)

Each phase leaves the repo in a buildable, testable state.

### Phase A · Foundation
- `git init`; commit current state as baseline.
- Remove tracked `dist/` directory; keep in `.gitignore` (already present).
- Remove self-referencing dependency (`"neobrutalistcomponents": "file:../neobrutalistcomponents/dist"`) from `package.json`.
- Delete the current `src/App.tsx` (it imports from the self-reference and will be replaced in Phase E).
- Add `LICENSE` (MIT).
- Replace `README.md` (current content is the default Vite template) with a real README: one-paragraph pitch, install command, minimal usage snippet, link to the landing URL, license line.

### Phase B · Theming foundation
- Create `src/lib/tokens.css` with base tokens and safe defaults.
- Create `src/lib/themes/{classic,tech,swiss,y2k}.css` satisfying the contract from §4.2.
- Create `src/lib/NeoProvider.tsx` exporting `NeoProvider`, `useTheme`, and the `NeoTheme` union.
- Tests for provider + hook.

### Phase C · Components
- Rewrite `Button.tsx` + `Button.css` against the API in §5.1. Implement all declared props (the current implementation only honors `variant`).
- Rewrite `Input.tsx` + `Input.css` against §5.2. Move from inline styles to CSS; add full a11y wiring.
- Write `Card.tsx` + `Card.css` (new component) with composable sub-components per §5.3.
- Co-located `*.test.tsx` for each.

### Phase D · Tests
- Install Vitest + RTL + axe-core; set up `vitest.config.ts`.
- Run tests locally. Coverage report generated.

### Phase E · Site
- Create `src/site/App.tsx` per §6. Site imports components via relative path `../lib` (no package-name self-import).
- Update `src/main.tsx` to mount `./site/App.tsx` and import the 4 theme CSS files upfront.
- Implement `ThemeSwitcher`, 3 showcase components, `CodeBlock` helper.
- Add `vite.site.config.ts`; confirm `npm run dev` renders the full landing.

### Phase F · Release
- Add 3 GitHub Actions workflows.
- Push to GitHub (new repo, public).
- Enable Pages. Verify deploy.
- `npm publish` at `0.1.0`.
